欠同事一週的文章：什麼是前端、後端與全端？

API 前端 後端 全端
 2026/03/16  次瀏覽  Share

"""

這篇文章，我其實欠公司一位同仁很久了。

本來只是很單純地說：「好，我再找時間跟你解釋什麼是前端、什麼是後端、什麼又叫全端。」結果這個「找時間」，一找就是一個禮拜。身為工程師，欠技術債已經夠可怕了，連文章債都欠，實在有點不好意思。

所以今天就來把這篇補上，而且盡量用最白話、最好理解的方式來說明。這篇不是要寫成教科書，也不是要端出一堆艱澀名詞把大家嚇跑，而是希望讓沒有技術背景的人，看完也能大概知道：

  • 前端是在做什麼
  • 後端是在做什麼
  • 全端又為什麼常常被講得很神

先講結論：你看到的，通常是前端；你看不到的，很多是後端

如果要用一句話先快速區分，我會這樣說：

跟畫面、瀏覽器、使用者操作直接有關的，多半是前端。

藏在伺服器、資料庫、商業邏輯後面的，多半是後端。

這樣講當然不是百分之百精準，但對入門理解非常夠用了。

很多人一聽到「前端」，直覺就是「喔，就是畫畫面的人」。這樣理解不能說錯，但其實只對了一半。因為前端除了處理「看得到的畫面」，也要處理很多「你看不到，但發生在瀏覽器裡的事情」。

而後端呢，通常更神秘一點，因為它大多待在伺服器裡默默工作。你平常不會直接看到它，但網站能不能正常運作、資料對不對、登入有沒有成功，很多都是後端在撐場。

什麼是前端？

前端，簡單來說，就是直接面對使用者的那一端。

只要是你打開網站後，在瀏覽器裡看到、點到、輸入到、互動到的東西，大多都屬於前端的範圍。像是：

  • 首頁長什麼樣子
  • 按鈕放哪裡
  • 表單怎麼輸入
  • 點擊之後畫面怎麼切換
  • 手機版畫面會不會跑版
  • 資料拿回來之後怎麼顯示在畫面上

這些都很直觀，因為「看得到」。

如果你把整個網站想成一間正在運作的店，前端有點像站在第一線接待客人的那個人。客人看到的第一印象、能不能順利互動、操作時會不會覺得舒服，很多都跟這個位置有關。

但前端有趣的地方就在於，它不只管看得到的部分，也會處理很多瀏覽器裡「看不到」的事情。

例如：為什麼你登入一次，瀏覽器就像認得你了？

你有沒有遇過這種情況？

你進一個網站，按下登入，輸入帳號密碼，成功登入後就可以開始使用。然後你重新整理頁面，咦，還是登入狀態。甚至你把分頁關掉，過一段時間再回來，有時候網站還記得你。

很多人會覺得這很神奇，好像瀏覽器突然有了記憶。

其實這背後沒有魔法，只有工程師。

通常流程會是這樣：

  1. 你在前端畫面輸入帳號密碼。
  2. 前端把這些資料送去後端。
  3. 後端驗證帳號密碼是不是正確。
  4. 如果正確，後端會回傳一個能代表「你已經登入」的資訊。
  5. 前端再把這個資訊暫存在瀏覽器裡。

這個暫存的位置，可能是 Cookie、Local Storage、Session Storage，或者其他瀏覽器端的機制。名字先不用急著背，入門先知道一件事就好：

瀏覽器之所以像是記得你，是因為前端把後端驗證後回傳的資訊，暫時存放在瀏覽器裡。

如果把這件事畫成一個小劇場，大概就像櫃台確認完你的身份之後，交給瀏覽器一張通行證，讓它下次看到你時知道：「喔，這位我認得。」

所以你看，前端不是只有畫面漂亮不漂亮而已，它還要負責管理很多使用者操作流程，以及瀏覽器內部的狀態。

也就是說，前端工程師不是只會排版。很多時候他還得處理：

  • 登入狀態怎麼保存
  • API 拿回來的資料怎麼整理
  • 按下按鈕後要不要跳提示
  • 資料還沒回來時要不要先顯示 loading
  • 權限不對時要不要擋畫面

這些事情雖然使用者不一定會注意到，但只要沒處理好，網站馬上就會很有個性，而且通常是那種讓人火大的個性。

什麼是後端？

如果說前端是站在第一線，直接面對使用者，那後端就是在後方撐住整個系統的那一群。

後端比較常見可以分成幾個部分：

  • Server
  • Database
  • 後端程式碼

因為這些東西大多不直接展示給使用者看，所以對非工程背景的人來說，後端常常會有一種神秘感。明明看不到，但它又真的存在，而且出了問題時，整個網站就會像失去靈魂一樣。

後端可以想像成什麼？

你可以想像一下，你現在正在用的這台電腦，其實也可以拿來當一台後端 Server。

假設今天你在這台電腦上：

  • 架了一個資料庫
  • 寫了一套 Web Server
  • 把後端程式跑起來

那這台電腦就可以扮演後端的角色。

前端只要知道這台機器提供哪些 API，就可以透過網路去跟它要資料、送資料、更新資料。

例如：

  • 我要登入，請幫我驗證帳密
  • 我要看會員資料，請把資料給我
  • 我要查商品列表，請把商品資料吐給我
  • 我要送出訂單，請幫我寫進資料庫

這些事情，通常就是後端在處理。

後端到底在忙什麼？

後端不是單純「把資料拿出來」而已，它常常還要做很多判斷與保護，例如：

  • 這個人有沒有登入
  • 這個人有沒有權限看這份資料
  • 傳進來的參數是不是合法
  • 資料要存到哪一張資料表
  • 如果資料衝突了該怎麼辦
  • 回傳給前端的格式要長什麼樣子

也就是說，後端常常是在處理規則、邏輯、資料與安全性。

如果把網站比喻成一家餐廳，前端像是外場，負責接待客人、送菜單、讓客人知道該怎麼點餐；後端就像內場廚房，負責真的把菜做出來、確認材料夠不夠、訂單有沒有問題。

這也是我很喜歡拿來解釋後端的比喻，因為它不只是神秘地待在後面，而是真的有一群人在忙著把規則、資料和流程穩穩接住。

你在外面看到餐點順順地送上桌，背後通常都不是奇蹟，而是一整套流程在運作。

前端跟後端怎麼合作？

講到這裡，通常下一個問題就是：

「那前端跟後端是怎麼溝通的？」

答案就是：透過 API。

你可以先把 API 想成一個「雙方約定好的溝通窗口」。

前端需要什麼資料，就去呼叫後端提供的 API。後端收到請求後，去做該做的事，再把結果回傳給前端。前端最後再把這些資料渲染到畫面上。

所以一個很常見的流程就是：

  1. 使用者在前端點了一個按鈕。
  2. 前端呼叫後端 API。
  3. 後端去查資料庫或做邏輯處理。
  4. 後端把結果用固定格式回傳。
  5. 前端收到後，把資料顯示在畫面上。

例如你打開一個訂單頁面：

  • 前端說：「我要訂單列表。」
  • 後端說：「好，我去資料庫查一下。」
  • 查完後，後端把 JSON 資料回傳。
  • 前端收到後，把它排成表格讓你看。

整個過程如果配合得好，使用者就會覺得網站很順。

如果配合得不好，就會出現各種經典場面，例如：

  • 前端說要 user_id，後端回 uid
  • 前端以為回傳的是陣列，結果後端回一個物件
  • 前端以為成功時會回 code: 200，結果後端回的是 status: ok

最後大家就會在會議室裡深情對望，然後開始說：「我以為你那邊會改。」

所以，API 溝通真的不是「有串到就好」，而是前後端要先講好規格。

API 規格，真的是前後端先講好的

這件事我很想特別講一下，因為很多剛接觸的人會以為 API 好像是後端自己寫一寫，前端再想辦法猜。

其實不是。

正常的開發流程裡，前後端通常會先約定好很多事情，例如：

  • 要用什麼 HTTP 方法，例如 GET、POST、PUT、DELETE
  • 請求要帶哪些參數
  • 參數名稱叫什麼
  • 回傳資料格式長什麼樣子
  • 成功時怎麼回
  • 失敗時怎麼回
  • 錯誤訊息要怎麼表示

這些規則如果沒有先談好，開發過程就很容易變成大型猜謎現場。

前端會猜：「你這個欄位是不是可能為空？」

後端會猜：「你那邊應該知道這個欄位不是每次都會回吧？」

猜到最後，專案進度就會像壞掉的跑步機，看起來很努力，實際上沒往前多少。

所以前後端的合作，除了寫程式本身，溝通規格也是很重要的一部分。

那什麼是全端？

講完前端與後端，全端就好理解很多了。

全端工程師，簡單來說，就是前端跟後端都能處理的人。

如果把場景再拉大一點，全端有點像那種不只會站前場，也敢走進後場，甚至能幫忙把兩邊節奏對起來的人。

他可能可以：

  • 寫前端畫面
  • 串接 API
  • 寫後端邏輯
  • 設計資料庫
  • 處理部署與基本維運

也就是從使用者打開網站，到資料真的被存進資料庫，中間整條鏈路他都大概看得懂，甚至都能動手做。

全端是不是等於什麼都要超強？

這裡也要幫全端平反一下。

很多人聽到全端，會以為那是一種神級生物，前端、後端、資料庫、Server、CI/CD、資安、UI/UX 全部點滿，晚上還順便修一下公司印表機。

現實通常不是這樣。

比較務實的理解是：

全端不是每一科都一定要滿分，而是你對整體流程都有概念，也能跨區域解問題。

有些全端偏前端，有些全端偏後端，這都很正常。

重點是，當一個功能要從畫面一路接到資料庫時，他不會只看自己那一小塊，而是能理解整體怎麼串起來。

這也是為什麼很多公司喜歡找全端，因為溝通成本通常會低一些，推進事情也更快一些。

當然，代價就是腦容量常常不夠用，昨天還在切版，今天就在查 SQL，明天可能又在看 Nginx 設定。某種程度上，全端工程師的日常，就是一種很有技術含量的精神分裂。

一個最簡單的例子：登入功能，到底誰在做什麼？

如果我們用「登入」這件事來拆開看，其實前端、後端、全端的分工會非常清楚。

前端負責的部分

  • 做出登入頁面
  • 讓使用者可以輸入帳號密碼
  • 按下登入按鈕後把資料送出去
  • 收到後端回應後，決定要跳成功、失敗，還是顯示錯誤訊息
  • 把登入成功後的資訊暫存在瀏覽器

後端負責的部分

  • 接收前端送來的帳號密碼
  • 驗證帳號密碼正不正確
  • 去資料庫查詢這個使用者
  • 判斷帳號狀態是否正常
  • 驗證成功後回傳登入憑證或相關資訊

全端會怎麼看這件事？

全端通常會從頭到尾一起看：

  • 前端頁面怎麼設計比較順
  • API 路由怎麼命名比較合理
  • 回傳格式怎麼定比較好串
  • Token 要怎麼存比較安全
  • 資料表該怎麼設計比較不容易出問題

所以全端的價值，很多時候不是「一個人做全部比較帥」，而是「一個人比較容易看懂整個系統怎麼運作」。

我自己的理解，總結成一句話

如果你今天只是想先快速抓住概念，我會這樣整理：

  • 前端：凡是跟畫面、瀏覽器、使用者互動，以及瀏覽器端狀態管理有關的，多半都算前端。
  • 後端：凡是跟 Server、資料庫、商業邏輯、驗證、資料處理有關的，多半都算後端。
  • 全端：能把前端跟後端串起來，理解整條流程，並能跨兩邊處理問題的人。

這個說法不是學術定義，但對大部分想入門的人來說，我覺得很夠用了。

結語：別再把工程師都叫做「會修電腦的」

寫到這裡，希望這位被我拖稿一週的同仁，看到這篇之後可以原諒我。

也希望其他看到這篇文章的朋友，以後在聽到前端、後端、全端時，不會再覺得那是一堆很遠很硬的名詞。

它其實沒有那麼玄。

前端，就是讓你看得到、點得到、用得到。

後端，就是讓這一切真的能運作，真的有資料，真的有邏輯。

全端，就是兩邊都懂，負責把整台車從方向盤一路接到引擎。

而如果再往上多看一層，還會碰到 Infra。

所謂的 Infra，其實就是 Infrastructure，也就是系統運作的基礎設施。白話一點講，就是除了前端、後端程式本身之外，還要去處理「這整個系統到底要跑在哪裡、怎麼連、怎麼部署、怎麼撐住」這些事情。

如果前端像舞台上的演員，後端像幕後實際把內容做出來的團隊，那 Infra 就更像是整個舞台後勤與總控室。觀眾不一定會看到他們，但燈光、線路、機器、節奏，只要哪裡出問題，整場演出就會立刻露餡。

例如：

  • Server 要怎麼配置
  • Database 要怎麼部署
  • Nginx 或 Web Server 要怎麼設定
  • Docker 要怎麼包服務
  • 網路怎麼通
  • Port 怎麼開
  • 反向代理怎麼設計
  • 整體系統架構要怎麼規劃才比較穩

這些東西，通常又會比單純的後端再多跨出去一點。因為它已經不只是寫商業邏輯，而是開始思考整個服務怎麼活下來、怎麼順順地跑、怎麼讓前後端可以穩定合作。

坦白說，這也剛好是我自己一直在累積的能力。

我雖然本業是後端出身，但我的技能其實不只是後端而已。我可以掌握前端與後端的整體串接，也能往上處理 Infra 相關的事情，包含 Server、部署、網路設計，甚至整個系統架構該怎麼規劃。

所以如果要用一句比較不害羞，但又算貼切的說法來形容我自己，我會說：

我不是只會寫某一端的工程師，而是能從前端、後端一路看到 Infra 與系統架構的人。

畢竟現在很多問題，真的不是只會寫 API 或只會切版就能解決。你常常會遇到的情況是：程式明明寫好了，但部署不起來；服務明明有跑，但網路不通；功能明明正常，但整體架構一大就開始出現效能或維運問題。

而能同時理解前端、後端、Infra、網路與系統架構的人，在這種時候就會特別有價值。因為他看到的不只是單一功能，而是整個系統如何從 0 到 1 被建立起來，並且穩定地活著。

至於工程師是不是都會修電腦？

嗯……這個問題就跟你問廚師會不會修瓦斯爐一樣，有些人會，但真的不是職稱上的必修科目。
