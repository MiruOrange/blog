一個計數器API的誕生：看似簡單卻暗藏玄機的Side Project

Side Project API DevOps Flask Docker Nginx Certbot 原子習慣

"""

最近完成了一個小小的 Side Project，為這個部落格打造了一個專屬的頁面計數器 API。回顧整個過程，我赫然發現，這段心路歷程的複雜與精彩程度，大概都能當作明年的鐵人賽主題了。

💡 起心動念：告別不穩定的免費服務

故事的開頭很單純。一直以來，我的部落格都是使用免費的第三方計數器服務。但天下沒有白吃的午餐，這個服務時好時壞，數據也常常在某次更新後就人間蒸發。身為一個有強迫症的工程師，看著數字歸零或無法顯示，實在是難以忍受。

於是，一個念頭油然而生：「乾脆自己架一個 API 來計數吧！」

這個想法聽起來很直接，但在我的部落格架構下卻有一個根本性的挑戰：我的網站是基於 GitHub Pages 的靜態頁面，這意味著我沒有任何可以執行後端程式的空間。要實現這個目標，等於是從零到一打造一個獨立的網路服務。

🗺️ 盤點與規劃：從零到一的建置藍圖

在下了這個「自幹」的決心後，我深吸一口氣，開始盤點所有需要完成的任務。不列不知道，一列嚇一跳，整個待辦清單遠比想像中來得長：

  1. ☁️ 申請雲端主機：在 Oracle Cloud 上申請一台 Always Free 等級的 VM。
  2. 🐳 環境容器化：在 VM 上安裝 Docker，讓未來的服務都能標準化、好管理。
  3. 🌐 建立網路門戶：啟動一個 Nginx 服務，作為所有流量的入口。
  4. 🏃 架設 GitLab Runner：在 VM 上安裝並註冊一個 GitLab Runner，為 CI/CD 做準備。
  5. 🚀 建置 CI/CD Pipeline：撰寫 .gitlab-ci.yml，設定自動化測試、打包、並部署到伺服器，方便每次更新就可以直接部署。
  6. 📦 準備數據倉庫：同樣用 Docker 啟動一個 PostgreSQL 資料庫服務。
  7. 🏷️ 擁有自己的門牌：購買一個專屬的網域名稱。
  8. 🔥 打通防火牆：設定 Oracle Cloud 的網路安全群組 (Firewall)，開放 80 和 443 埠。
  9. ➡️ 建立網址指向：在 DNS 服務商那邊設定 A 紀錄，將我的網址指向 VM 的公開 IP。
  10. ⚙️ 配置反向代理：調整 Nginx 設定，增加 Certbot 驗證用的路由規則。
  11. 🔒 取得安全憑證：撰寫 Shell Script，自動化 Certbot 容器的啟動與驗證，順利拿到免費的 SSL 憑證。
  12. 🧪 開發 API 雛形：啟一個 Flask 服務，並在 VM 本地端測試確定能成功訪問。
  13. 📈 設計計數功能：為 Flask 服務增加計數器的 Route，並用 Postman 徹底測試請求與回應。
  14. 🗂️ 建立資料表：在 PostgreSQL 資料庫上建立專門用來儲存計數資訊的 Table。
  15. 🔄 串接資料庫：在 Flask 中增加存取資料庫的函式，並測試 CRUD 功能是否正常。
  16. ✅ 完成 API 邏輯：將計數器 API 的所有邏輯完成，並精心設計 Response 的 JSON 格式。
  17. 💻 前端串接：回到部落格的程式碼，調整前端 JavaScript，非同步地呼叫我全新的 API。
  18. 🎉 大功告成！

📚 知識的廣度：一個都不能少

這個看似簡單的計數器，背後卻牽涉了相當廣泛的知識領域。每一個環節都是一門專業，缺一不可：

完成這個小小的 API，竟然需要橫跨這麼多領域的技能。有時候我會開玩笑地想，這樣的分享會不會直接勸退一堆想當工程師的朋友呢？（笑）

🎯 總結：工程師不只寫 Code，更要學會規劃與執行

這個專案最大的收穫，不僅僅是那個穩定運作的計數器，更是再次體悟到：「想當一個稱職的工程師，除了鑽研技術，更要學會規劃與執行計畫。」

面對這份長長的待辦清單，如果想著要一次做完，恐怕永遠沒有開始的勇氣。這次我採用了「原子習慣」的精神，將任務切分成每天一點點、一點點的進度。我規定自己每天至少要完成一小步，哪怕只是「建立一個資料夾」這種微不足道的小事。

透過身體力行，讓「持續推進專案」成為一種習慣。最終，這個看似龐大的工程，就在每天的累積下一步步完成了。

希望這次的經驗分享，能帶給螢幕前的你一些啟發。

"""
