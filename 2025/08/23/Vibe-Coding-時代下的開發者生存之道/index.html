Vibe Coding 時代下的開發者生存之道

AI Vibe Coding 職涯規劃

"""

「那是一個最好的時代，也是一個最壞的時代。」

狄更斯在《雙城記》的開頭這樣寫道。我想，這句話完美地詮釋了當下我們程式開發人員所處的環境。AI 技術浪潮席捲而來，Vibe Coding (或稱 AI 輔助開發) 成為了我們手中最強大、也最具爭議性的工具。對於那些願意擁抱並持續學習 AI 工具的開發者來說，這無疑是一大福音；然而，對於安於現狀的人，「開發人員即將被取代」的焦慮也從未如此真實。

但我認為，真正會被取代的，從來都不是開發人員這個職位，而是那些「不願意掌握 AI 工具的人」。

一場來自 PM 部門的震撼教育

最近，我參加了公司內部一場關於 Vibe Coding 的講座，而最讓我驚訝的是，主講的竟然是完全沒有程式背景的 PM 部門。

他們分享了導入 AI 前後的開發流程變化，讓我感觸良多。

過去的開發流程

在過去，一個功能的誕生需要跨越多個部門，流程冗長而繁瑣：

graph LR
    A[PM 團隊撰寫需求] --> B[與開發 Team PM 溝通需求]
    B --> C[UI/UX 設計團隊繪製介面]
    C --> D[RD 研發團隊進行開發]
    D --> E[Infra 團隊協助部署與維運]
    E --> F[QC 團隊進行測試]
graph TD
    A[PM 團隊撰寫需求] --> B[與開發 Team PM 溝通需求]
    B --> C[UI/UX 設計團隊繪製介面]
    C --> D[RD 研發團隊進行開發]
    D --> E[Infra 團隊協助部署與維運]
    E --> F[QC 團隊進行測試]

導入 Vibe Coding 後的流程

而現在，PM 們利用 Vibe Coding 工具，整個流程被極大地簡化了：

graph LR
    A[PM 團隊撰寫需求] --> B[透過 Vibe Coding 直接生成應用]
    B --> C[RD 團隊審查架構與資安問題]
    C --> D[Infra 團隊協助部署與維運]
graph TD
    A[PM 團隊撰寫需求] --> B[透過 Vibe Coding 直接生成應用]
    B --> C[RD 團隊審查架構與資安問題]
    C --> D[Infra 團隊協助部署與維運]

從上面的流程圖可以清楚看到，UI/UX 設計師和前端開發者的部分工作，在許多情境下幾乎被完全跳過了。PM 們透過精準的 Prompt，就能直接產出一個功能完整的原型。這對前端和 UI/UX 設計師來說，無疑是一個巨大的衝擊。

後端工程師的獨特價值

看到這樣的轉變，我不禁慶幸自己當初選擇了後端工程師這條路。身為一名後端，我不僅僅是「寫程式的人」。我的價值體現在：

  • 系統架構設計：AI 或許能寫出單一功能的程式碼，但它無法從全局思考，設計出一個高可用、高擴展性的系統架構。
  • 資料庫規劃：從 Schema 設計、索引優化到效能調校，這些都需要深入的領域知識與經驗。
  • 基礎設施 (Infra) 與網路知識：我比一般的後端更了解伺服器、網路、部署流程，這讓我在系統上線的最後一哩路扮演了關鍵角色。
  • 持續精進的 AI 技能：結合我原有的技術棧，我能利用 AI 更快地解決問題、搭建服務，甚至找出 AI 生成程式碼中的潛在風險。

這些跨領域的綜合能力，讓我成為一個難以被輕易「取代」的角色。

我們還需要學習程式語言嗎？

絕對需要！這是我最堅定的看法。

AI 終究只是一個工具。如果它產出的程式碼你完全看不懂，你要如何判斷它寫得好不好？有沒有邏輯漏洞？會不會有效能問題？

更可怕的是資安問題。如果你連如何保護敏感資訊 (例如 API Token、資料庫密碼) 的基本概念都沒有，在使用 Vibe Coding 的過程中，這些關鍵資訊可能已經在不知不覺中被洩漏給了 AI 模型。這就像是請一個你不完全信任的陌生人來幫你蓋房子，卻任由他隨意進出你的金庫，這是何等危險！

結語

回到開頭那句話：「那是一個最好的時代，也是一個最壞的時代。」

AI 帶來了前所未有的效率與可能性，這是最好的部分。但同時，它也無情地淘汰那些停止學習、無法適應變化的角色，這是最壞的部分。

未來屬於那些懂得善用工具、並結合自身專業領域知識，創造出更大價值的人。你，跟上潮流了嗎？

"""
